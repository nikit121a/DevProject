                                    XAML
                                    [XAML]
                                    <dx:WindowedDocumentUIService WindowType="dxwui:WinUIDialogWindow" x:Name="FormsDocumentManagerService" YieldToParent="True">
            <dx:WindowedDocumentUIService.WindowStyle>
                <Style TargetType="dxwui:WinUIDialogWindow">
                    <Setter Property="ShowTitle" Value="True" />

            </dx:WindowedDocumentUIService.WindowStyle>
        </dx:WindowedDocumentUIService>

                                    C#
                                    [C#]
                                     public void OpenShipmentForms()
        {
            if (CanSave())
                Save();

            var unitOfWorkFactory = UnitOfWorkSource.GetUnitOfWorkFactory();
            var unitOfWork = unitOfWorkFactory.CreateUnitOfWork();
            var entities = unitOfWork.ShipmentForms;
            var selectedEntity = entities.Where(s => s.ShipmentId == this.Entity.ShipmentId).FirstOrDefault<ShipmentForm>();

            if (selectedEntity != null)
            {
                this.GetRequiredService<IDocumentManagerService>("FormsDocumentManagerService").ShowExistingEntityDocument<ShipmentForm, int>(this, this.Entity.ShipmentId);
            }
            else
            {
                Action<ShipmentForm> initializer = x =>
                {
                    x.ShipmentId = Entity.ShipmentId;
                };

                this.GetRequiredService<IDocumentManagerService>("FormsDocumentManagerService").CreateDocument("ShipmentFormView", null, initializer, this).Show();
            }

        }

                                  Hi Steve,

                                  The approach you demonstrated, i.e., the use of the ShowExistingEntityDocument/CreateDocument method, is generally correct. I have reproduced the behavior you mentioned - the dialog cannot be closed - and see that it occurs because documents with DestroyOnClose=false are supposed to be hidden rather than closed. While this works with no issues in regular windows, WinUIDialogWindow displays its content in the window's Adorner, and it cannot be hidden in a regular manner. Taking this into account, I confirm that setting the DestroyOnClose property to true is the correct approach in this scenario.

                                  As for the second issue - the dialog is displayed empty - I have not managed to reproduce it on my side and would like to ask you to check the following points:

                                • Kirill (DevExpress Support) 06.08.2018

                                  Hello,
                                  The main goal, in this case, is to track WinUIDialogWindow's Close and Closing events invocations and their count. The Close and Closing events are events of the WinUIDialogWindow element which are derived from the standard Window control. So, you can subscribe to them in your WindowStyle property declaration using the EventToCommand behavior in the following manner:

                                  [XAML]
                                  <dx:WindowedDocumentUIService.WindowStyle>
    <Style TargetType="dxwui:WinUIDialogWindow">
        <Setter Property="ShowTitle" Value="True" />
        <Setter Property="dxmvvm:Interaction.BehaviorsTemplate">
            <Setter.Value>
                <DataTemplate>
                    <ItemsControl>
                        <dxmvvm:EventToCommand EventName="Closing" Command="{Binding ClosingCommand}"/>
                        <dxmvvm:EventToCommand EventName="Closed" Command="{Binding ClosedCommand}"/>
                    </ItemsControl>
                </DataTemplate>
            </Setter.Value>
        </Setter>
    </Style>
</dx:WindowedDocumentUIService.WindowStyle>

                                  The "ClosingCommand" and "ClosedCommand" commands are defined at the ViewModel level:

                                  [C#]
                                  public ICommand ClosingCommand { get { return new DelegateCommand(() => Closing()); } }
public ICommand ClosedCommand { get { return new DelegateCommand(() => Closed()); } }

private void Closed() {
    //
}

private void Closing() {
    //
}

                                  As we need to collect these events invocation order and count, I suggest that you define and use custom data (see the Add Custom Data to Reports article for details):

                                  [C#]
                                  private void Closing() {
    LogifyAlert client = LogifyAlert.Instance;
    client.CustomData[DateTime.Now] = "Closing event is called";
}

                                  The same custom data creation logic should be placed to the Close and SaveAndClose commands' methods. As the SaveAndClose command usually calls the Close command, it should call your modified version of the Close command which creates the described custom data.

                                  Finally, it's necessary to call the LogifyAlert.Instance.Send method to send us a report with this custom data. It's necessary to call this method in a separate thread with a delay (or using a Dispatcher) to send us this report after the all the custom data is collected. You can invoke the report sending logic in the Closed event handler since this event is raised after all other methods and events.

                                  Apparently, it lies in WinUIDialogWindow's nature. When WinUIDialogWindow is used in non-dialog mode, it is not actually shown. It detaches its content and displays it in the window's Adorner. After that, it cannot be hidden in a regular manner. Our developers are researching this behavior and will try to find ways to fix it. At the moment, I cannot provide you with information as to when it can happen.

                                  As a solution, we recommend you use your WinUIDialogWindow in dialog mode. For this, set the WindowedDocumentUIService.DocumentShowMode property to Dialog. Please take special note that in this case the DestroyOnClose property in the CreateDocument method should be set to True.
