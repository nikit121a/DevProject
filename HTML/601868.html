                                PDFViewer - Problem printing PDF's - prints for a while and then stops

                                Tags:
                                • .NET,
                                • WinForms,
                                • PDF Viewer
                                RCS 01.31.2018
                                  • We are experiencing a problem with printing from the pdf viewer.  Users will be able to print for a while and then they will lose the ability to print unless they close the application and restart.  After restarting they are able to print again for a while. I attempted to enable legacy printing and that did not help.  I have also gone through our code and tried to make sure the PDFViewer was being disposed when the form or control it is is used on is closed/disposed.

                                    When the problem occurs no errors are generated.  The print dialog appears briefly and shows the spooling message. Only page 1 is displayed before the dialog closes.  If the print queue manager window is open while this is happening, you can see the print job appears briefly and then quickly disappears.

                                    Printers:  Kyocera FS-2100DN Driver version: KX Print Driver v7.3.1207

                                    We have also seen this on an HP printer.

                                • RCS 02.02.2018
                                  The added disposal code seems to have corrected the issue.  Basically, what we did was add code to make sure the PDF Viewer control was explicitly disposed.   

                                  For example, we have the pdf viewer embedded in a form and previously did not have code in place in the form to explicitly call the Dispose method on the pdf viewer.  The form was being disposed, but the DE pdf viewer control did not have the disposed method called on it.  We added logic in the form to call Dispose on the pdf viewer when the form is closed/disposed.  We also have a user control that contains the pdf viewer.  For our control we added code to have the viewer's Dispose method called when our control was being disposed.  We also made sure that any other code that used our user control would call Dispose on the user control so that the pdf viewer would have Dispose called on it, too.

                                  Since making these changes, we have not had any printing issues.

                                • Yulia (DevExpress Support) 06.28.2018

                                  Hello,


                                  >>We also periodically see blank pages printed. <<
                                  This behavior usually occurs if there is no enough memory to render a page. The best approach to diagnose memory issues is to profile the application using any Memory Profiler tool. Usually, such tools show which objects are not disposed of, and you can determine the place in your/our code, which causes memory to leak.

                                  Would you please check if there are any user handled exception when the printing issue occurs? To check this, turn on the  "Common Language Runtime Exceptions" item in the Debug | Exceptions... window in Visual Studio. Please refer to the How to obtain the exception's call stack KB article for more detailed instructions. Collect all exceptions and their details to a file and provide us with it. 

                                  I have researched the code you provided and found that you dispose of the PdfViewer control in the Form.Closing event handler manually. The control should be automatically collected by the garbage collector when the form is disposed of. So, remove this code and let us know if the printing issue occurs after this.

                                • Yulia (DevExpress Support) 06.29.2018

                                  Thank you for your clarification. Would you please check which build of Windows 10 is installed on the client PC (How to check what Windows 10 build you are on in two easy steps)? Is the PdfPrinterSettings.EnableLegacyPrinting option enabled in your project?


                                  Recently, we found a Microsoft bug with printing documents with .OTF fonts on Windows 10 (Build 1803) OS (the Print Page dialog is shown, however, the document is not printed). You can find more information on this issue on the Microsoft forum: Update 1803 broke printing from office documents using .otf fonts.
                                  The aforementioned bug was reported for MS Office Word. However, the problem will appear with any application that renders OpenType fonts using GDI. 

                                  You can check if the PDF documents that were not printed contain .OTF fonts in Adobe Reader. Open the Document Properties window, the Fonts tab page and check if there are Type 1 embedded fonts in the document (see the attached screenshot).


                                  This problem does not occur with the PdfViewer's legacy (previous) printing mode. This mode can be turned on using the PdfPrinterSettings.EnableLegacyPrinting property with the code below:

                                  [C#]
                                    PrinterSettings printerSettings = new PrinterSettings();
  printerSettings.PrinterName = yourPrinterName;
   PdfPrinterSettings pdfPrinterSettings = new PdfPrinterSettings(printerSettings);
  pdfPrinterSettings.EnableLegacyPrinting = true;
  pdfViewer1.Print(pdfPrinterSettings);

                                   

                                  When the EnableLegacyPrinting option is enabled, we manually generate and rasterize images and send these images to a printer. This operation is slower than the new printing mechanism and allocates more memory while printing the document. So, blank pages can be produced while printing large documents in this mode. However, with the old mechanism, there is no direct dependence on a printer driver.
