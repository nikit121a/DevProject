                                Andrey Marten (DevExpress Support) 02.13.2018


                                Hello Khabir,

                                Having worked with WPF for a long time, I agree that the abundance of events can sometimes be confusing. In the meantime, all events that you mentioned are the integral part of the WPF framework and are exposed by every WPF control (see: LostFocus, LostKeyboardFocus, LostMouseCapture, IsKeyboardFocusWithinChanged). 

                                I've prepared a simple sample to illustrate how the standard TextBox and TextEdit behaves under the same conditions. Please refer to the attached screencast and dxSample1.zip.

                                As you can see, the LostFocus event is raised in the TextBox when I close Window and this behavior is not specific only to our editors. Moreover, when I focus TextBox and TextEdit in order, you can see that the behavior of the LostFocus and LostKeyboardFocus methods is exactly the same.

                                The only difference that I found is that the LostFocus and LostKeyboardFocus events are raised in TextEdit when I focus it by using the Tab key. This behavior is determined by the fact that TextEdit uses TextBox inside and when TextEdit is focused, it focuses its internal TextBox. As a result, TextEdit loses focus itself and corresponding events are raised. This behavior is expected for WPF controls that use other controls inside. Test, for example, the standard DatePicker control and its LostFocus and LostKeyboardFocus events. They are raised several times when its popup is opened (see also: WPF DatePicker LostFocus fires seven times).

                                The IsKeyboardFocuseWithin property, in turn, indicates that a certain element or one of its internal elements are focused. That is why the use of the IsKeyboardFocusedWithinChanged event may help in this case. You can also try handling Window's Closing event and set a certain flag, which will disable validation in the standard LostFocus/LostKeyboardFocus/IskeyboardFocusedWithinChanged event handlers. 

                                At the same time, none of these events were specially provided in our controls for validation purposes, though they definitely can be used for this purpose. Our editors provide a special Validate event, which allows you to use custom validation logic for a value entered by an end-user before this value is posted to EditValue. I suggest that you review the following documentation article to learn more about this event: BaseEdit.Validate Event. Perhaps, the use of this event will be more suitable in your application. 

                                I've prepared another sample showing how to use the Closing event to prevent custom validation logic in the standard focus events and illustrating how the Validate event can be used. Please refer to the attached dxSample2.zip file.

                                Should you have additional questions or face any difficulties using this event in your application, please feel free to contact us. More information about the behavior that you wish to achieve in your application will help us find a more suitable solution. 

                                See also: 
                                Input Validation Overview
                                Input Validation Types

                                Thanks,
                                Andrey

                                • Andrey Marten (DevExpress Support) 03.06.2018


                                  Hi Khabir,

                                  I've discussed your use case with our developers to determine if the behavior that you described can be implemented at the TextEdit level. Currently, if the InvalidValueBehavior property is set to 'WaitForValidValue', the editor locks the mouse input for all controls in its parent window. To get the parent window, the editor finds the root element in the visual tree using the standard VisualTreeHelper.GetParent method.

                                  In your sample project, you are using MDI panels to display forms with editors. MDI panels are not separate windows, they are a part of the main application window. This is why editors lock the mouse input for the entire application.

                                  I understand that you want to lock the mouse input only in the form where editors are declared. However, editors have no knowledge of your XAML files and project structure. They can only get elements in the visual tree to determine where they reside. Even if editors find that one of their parents is a UserControl, it still does not mean that this UserControl represents a form in your application. People often use UserControls to reduce the amount of duplicate code, and multiple UserControls can exist in a single form.

                                  Another point that you mentioned before is that you still want to be able to click the OK and Cancel buttons when an editor contains an invalid value. Since the OK and Cancel buttons reside in the same container as editors, editors can not possibly know that clicks on these buttons should be allowed.

                                  Having discussed your case with the team, I found that the best way to implement this custom validation logic is to set the InvalidValueBehavior property to 'AllowLeaveEditor' and manually handle mouse events for your form to lock input. I've attached a sample project (dxSample.zip) that demonstrates this approach. 

                                  Note that with this solution, you will not be able to lock the keyboard focus in an invalid editor in one form and enter values in an editor in another form. WPF does not allow locking the keyboard focus in two places in the same window. If you wish to keep two focused editors in two forms, display your forms using real windows instead of MDI panels. I've made another sample project (dxSample_MultipleWindows.zip) to demonstrate this approach.

                                  P.S. As for the Message Box, the Validate event handler is not the best place to show messages. It is expected that the Validate event fires more than once in certain scenarios, since the value synchronization logic in editors can be triggered by multiple factors (Binding updates, keyboard input, focus change, etc.). If you want to show the Message Box when a user tries to leave the editor, do this using mouse events (e.g., the ShouldHandleMouseEvents method in the sample).

                                  Thanks,
                                  Andrey
